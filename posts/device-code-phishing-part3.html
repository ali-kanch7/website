What Actually Works — and the Setting Almost Nobody Has Touched

The good news first: there's a real, well-documented fix for this, and Microsoft has even started turning part of it on by default. The honest news: most organizations still don't know it exists, let alone whether it's working for them.

Here's a genuinely good piece of news to start this part with, because the first two parts of this series spent a lot of time on the bad news: this isn't an unsolvable problem. Microsoft, to its credit, looked at this exact attack pattern and built a direct fix for it — not a workaround, not a "be more careful" suggestion, an actual setting that closes the door. The catch, and it's a real one, is that this fix isn't switched on everywhere by default yet, and most organizations have simply never gone looking for it, because nobody told them it existed.

So this part is the practical half of the series — what the fix actually is, how to turn it on, and the handful of ways even a correctly intentioned setup quietly fails anyway.

The Direct Fix

Microsoft Already Built the Off Switch

Remember the actual mechanism from Part 1: the entire attack depends on a feature called device code flow existing and being reachable by anyone. Take that feature away from accounts that don't genuinely need it, and the attack has nowhere left to go — there's no code to enter, because no code can be generated against your tenant in the first place. This isn't a clever workaround. It's closing the actual door the attacker walks through.

Microsoft Entra ID — the identity system behind Microsoft 365 — lets administrators do exactly this through a feature called Conditional Access. The setting that matters here is specifically labeled "Authentication flows," and inside it sits a toggle for device code flow itself. Block it, and any attempt to use that flow against your organization simply fails, for everyone the policy covers.

The actual configuration path, in Microsoft's own admin center
1. Sign in to the Entra admin center as a Conditional Access Administrator
2. Go to Protection → Conditional Access → Policies → New policy
3. Under Assignments, include all users — except your emergency "break-glass" accounts
4. Under Target resources, select all resources
5. Under Conditions → Authentication Flows, switch on Device code flow
6. Under Grant, choose Block access
7. Save it in Report-only mode first — confirm nothing legitimate breaks — then switch it to On

That last step matters more than it looks. Running the policy in "report-only" first means it watches and logs what would have happened without actually blocking anything yet — giving you a chance to discover, before you flip the switch for real, whether some printer, conference room display, or script genuinely depends on this flow. Skip that step and you risk locking out a legitimate piece of office equipment along with every attacker, on the same afternoon.

The encouraging part of this story

Microsoft didn't just publish a guide and leave it to individual administrators to find. Starting in February 2025, they began automatically rolling out a pre-built, Microsoft-managed version of exactly this policy to eligible organizations — switched on by default for tenants that show no recent legitimate use of device code flow. If your organization has the right licensing and hasn't actively gone out of its way to disable it, there's a real chance some of this protection is already quietly running in the background, without anyone having configured it by hand.

That's worth checking rather than assuming, though — head into Conditional Access and look for a policy whose "Created by" column says Microsoft rather than a person's name. If it's there, it's worth understanding rather than ignoring; if it's been switched off, that's worth asking why.

The Second Layer

What Happens If a Token Gets Stolen Anyway

Blocking the flow outright is the strongest move, but it's not always possible everywhere — some organizations genuinely do rely on device code flow for things like conference room displays, command-line developer tools, or specific older hardware, and blocking it completely would break real work. For those cases, and as a sound second layer even where blocking is possible, there's a feature worth knowing about called Continuous Access Evaluation, usually shortened to CAE.

The normal way access tokens behave is patient, almost to a fault: once issued, a token is typically trusted for up to an hour before anything rechecks it, which is exactly the kind of quiet window an attacker holding a stolen token benefits from. CAE changes that rhythm. Instead of waiting for the clock to run out, certain critical signals — an account getting disabled, a password reset, a user newly flagged as high-risk, a sign-in suddenly coming from a wildly different location — get evaluated in something close to real time, and access can be cut off within minutes rather than waiting for the token's natural expiry.

This doesn't stop the initial theft. What it does is shrink the window an attacker actually gets to use what they stole, from "however long the token happens to live" down to "however long it takes your systems to notice something's wrong." For a security team that's actually watching, that's the difference between an attacker quietly working for a day and an attacker getting cut off within minutes of anything looking unusual.

Where Good Setups Still Slip

The Quiet Ways This Protection Fails Anyway

It would be tidy to end here and say "block the flow, done." Real deployments are messier than that, and there are a few specific, well-documented ways a seemingly correct setup still leaves a gap open. Worth knowing these now, rather than discovering them the hard way later.

  • ⚠
    A token issued before the block went live can outlive the block itself

    Conditional Access is checked at the moment a token is issued, not continuously after the fact. If someone already holds a valid token from before your policy went into effect, simply turning the policy on afterward won't retroactively revoke it — there are documented real cases of administrators seeing a "successful device code sign-in" logged even with a global block supposedly in place, purely because an old token was still being reused. The fix is direct but easy to forget: when you tighten a policy like this, also revoke existing sessions for the accounts it covers.

  • ⚠
    Exceptions tend to widen quietly over time

    Almost every real rollout needs a short exceptions list — a conference room screen here, a CLI tool there. Each exception is reasonable on its own. The risk is that nobody revisits that list six months later, and what started as two narrowly justified exceptions slowly becomes a forgotten standing gap that nobody remembers granting.

  • ⚠
    Service accounts and automated scripts don't follow the same rules

    Conditional Access policies aimed at people don't automatically apply to non-human accounts — the service accounts and scripts running quietly in the background. If your organization uses these for any automated task, they need their own, separately considered policy; assuming the "all users" rule already covers them is a common and understandable mistake.

The Human Layer

The One Thing Worth Teaching People, Without Overloading Them

Technical controls matter most here, but there's a single, simple thing worth telling anyone in an organization, precisely because it costs nothing to explain and doesn't require understanding OAuth at all: a legitimate reason to enter a code on a Microsoft sign-in page almost always involves a device sitting right in front of you — a new TV, a conference room system being set up, a printer being configured. If a code shows up instead through an email, a chat message, or a phone call, asking you to go enter it somewhere, with no actual device anywhere nearby that's waiting for it, that mismatch alone is reason enough to stop and check with IT before typing anything in.

That's a far more durable lesson than "watch out for suspicious links," because it doesn't depend on spotting a fake website — there isn't one to spot. It depends on noticing a missing device, which is a much simpler thing for a non-technical person to actually catch in the moment.

The fix for a trick that works by feeling routine isn't more suspicion. It's one specific question, asked at the right moment: where's the device that's supposed to be asking me this?

— the one habit worth actually teaching

Pull the three parts of this series together and a pattern emerges that goes well beyond this one specific attack. A genuinely useful, well-intentioned feature — built to make life easier for devices that can't type — got quietly repurposed by people who realized it never checked who was actually asking. The defenses everyone already trusted weren't broken by it; they simply weren't standing at the door it walked through. And the fix, once you know where to look, turns out to be a setting that's been sitting in plain sight inside Microsoft's own admin console, switched off for most of the world by default, on by default for a fraction of organizations, and entirely unknown to almost everyone in between.

None of that is really a story about Microsoft, or even about this one flow specifically. It's a story about how every system, however carefully built, eventually meets someone who reads the fine print of how it actually works rather than how it's supposed to feel — and finds the gap sitting quietly in the space between those two things.
