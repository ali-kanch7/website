Linting Infrastructure: Catching VLAN Leaks

VLAN leakage is the digital equivalent of a leaky pipe in a skyscraper. You think your tenants are isolated, but somewhere in the walls, the plumbing is crossed. When that happens, traffic meant for a secure segment spills into an untrusted one. It’s a silent, catastrophic failure.

In modern infrastructure, we don't build networks by hand anymore. We write code. But if your Infrastructure-as-Code (IaC) has a bug, you aren't just deploying a server—you’re deploying a security hole at scale. To stop this, we need to move beyond simple syntax checks and start treating network topology as a mathematical constraint.

The Guardrail Pattern

Think of your CI/CD pipeline as a high-speed assembly line. If you wait until the end to inspect the product, you’ve already wasted time and money. We need to install guardrails.

This architecture relies on three layers:

  • The Linter: Your first line of defense. It parses your Terraform or Kubernetes manifests to find illegal VLAN IDs or missing isolation tags.
  • The Evaluator: This is the brain. It queries your "Source of Truth"—like NetBox—to see if the requested network change actually makes sense in the real world.
  • The Gatekeeper: The CI/CD step that slams the door shut if the policy is violated. No pass, no deploy.

Moving to Semantic Validation

Most linters are just spellcheckers. They tell you if you missed a bracket. That’s not enough for network security. You need semantic validation.

If a developer opens a pull request that defines a new subnet, the linter should ask: "Does this CIDR block overlap with existing production zones?" If the answer is yes, the build fails immediately. You aren't just checking if the code is valid; you're checking if the infrastructure is safe.

Tools like Open Policy Agent (OPA) are perfect for this. You write a Rego policy that says: "Deny any resource with a VLAN ID between 100 and 199 unless the environment tag is explicitly set to prod-restricted." It’s declarative, versioned, and impossible to ignore.

The Challenges of Reality

It sounds great on paper, but reality is messy. The biggest trap is the "Stale State" problem. If your linter relies on a local copy of your network map, it’s already lying to you. Always query your live IPAM API during the pipeline. If the data isn't live, it isn't accurate.

Also, don't try to boil the ocean. If your infrastructure repo is massive, don't re-scan the entire network for every commit. Use incremental linting to check only what changed. Keep the pipeline fast, or developers will find a way to bypass it.

Summary

VLAN leakage isn't a networking problem; it’s a deployment problem. By embedding network constraints into your CI/CD pipeline, you stop mistakes before they become breaches. Treat your network policies as code, automate the validation, and let the machines handle the guardrails.

Stop relying on manual reviews. If it can be defined in code, it can be enforced by a machine.
